# Заявки в ремонтную службу

Небольшое веб-приложение для приёма и обработки заявок в ремонтную службу (диспетчер и мастера).

## Запуск (Docker Compose, предпочтительно)

Из корня репозитория:

```bash
docker compose up --build
```

Приложение будет доступно по адресу **http://localhost:8000**.

При первом запуске выполняются миграции и сиды. Повторный запуск не дублирует заявки (сиды проверяют наличие данных).

## Запуск без Docker

Требования: PHP 8.2+, Composer, расширение SQLite.

```bash
cd repair-app
cp .env.example .env
composer install
php artisan key:generate
touch database/database.sqlite
php artisan migrate
php artisan db:seed
php artisan serve
```

Откройте http://localhost:8000 .

## Страницы

1. **Создание заявки** — `/requests/create` (без входа): форма (клиент, телефон, адрес, описание). После создания статус заявки — `new`.
2. **Панель диспетчера** — после входа под диспетчером: список заявок, фильтр по статусу, назначение мастера (статус `assigned`), отмена заявки (`canceled`).
3. **Панель мастера** — после входа под мастером: список заявок, назначенных на этого мастера; кнопки «Взять в работу» (`assigned` → `in_progress`) и «Завершить» (`in_progress` → `done`).

## Проверка «гонки» (параллельное «Взять в работу»)

Действие «Взять в работу» сделано безопасным при параллельных запросах: используется блокировка строки в транзакции (`lockForUpdate()`). Один запрос успешен (200), второй получает **409 Conflict** с сообщением, что заявка уже взята.

### Вариант 1: два терминала с curl

1. Запустите приложение и войдите в браузере под мастером (master1@repair.local). Убедитесь, что есть заявка со статусом «assigned», назначенная на этого мастера (например, создайте её через панель диспетчера или используйте сидовую заявку с id=2).
2. Получите cookie сессии: в DevTools (F12) → Application → Cookies → скопируйте значение `laravel_session` (и при использовании — `XSRF-TOKEN`).
3. В **первом терминале** выполните (подставьте `REQUEST_ID`, `BASE_URL`, cookie и при необходимости CSRF):
   ```bash
   curl -X POST "http://localhost:8000/api/requests/take" \
     -H "Content-Type: application/json" \
     -H "Cookie: laravel_session=ВАШ_SESSION" \
     -H "X-XSRF-TOKEN: ВАШ_CSRF" \
     -d '{"request_id": REQUEST_ID}'
   ```
4. Одновременно (или почти одновременно) в **втором терминале** выполните тот же запрос с тем же `request_id`.
5. Ожидаемый результат: один ответ **200** с сообщением об успехе, второй — **409** с текстом вроде «Заявка уже взята в работу или недоступна для взятия».

### Вариант 2: скрипт race_test.sh

Из корня репозитория (или из каталога со скриптом):

```bash
chmod +x race_test.sh
./race_test.sh
```

Скрипт ожидает, что приложение доступно на `http://localhost:8000`. Он логинится под мастером, находит заявку в статусе `assigned` и отправляет два параллельных POST на «взять в работу». В выводе должны быть один успешный ответ (200) и один отказ (409). Подробности — в комментариях внутри скрипта и в README.

## Автотесты

Локально (нужен PHP 8.2 с расширением SQLite):

```bash
cd repair-app
composer install
php artisan test
```

Через Docker:

```bash
docker compose run --rm app php artisan test
```

Минимум два теста: создание заявки (форма и сохранение со статусом `new`) и проверка, что повторный вызов «взять в работу» для той же заявки возвращает отказ (без «поломки» заявки).
