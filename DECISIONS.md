# DECISIONS.md — ключевые решения

1. **Стек: Laravel 12 (PHP 8.2), SQLite**  
   Выбраны для быстрого локального запуска, наличия миграций, сидов и встроенной аутентификации. SQLite не требует отдельного сервера БД при запуске через Docker или `php artisan serve`.

2. **Роли в одной таблице `users`**  
   Роль хранится в поле `role` (`dispatcher` / `master`). Отдельные таблицы или полиморфные роли не вводились, чтобы упростить сиды и проверку доступа.

3. **Защита от гонки при «Взять в работу»**  
   В сервисе `RepairRequestService::takeInWork()` используется транзакция и `lockForUpdate()` по строке заявки. Только один параллельный запрос переводит заявку из `assigned` в `in_progress`; второй получает 409 Conflict с понятным сообщением. Это гарантирует, что заявка не «ломается» при одновременных запросах.

4. **API «взять в работу» в веб-маршрутах**  
   Эндпоинт POST `/api/requests/take` зарегистрирован в группе `web` с middleware `auth`, чтобы проверку гонки можно было делать через curl с cookie сессии (логин в браузере, затем два параллельных запроса с тем же cookie).

5. **Сиды идут после миграций при первом запуске**  
   В Docker Compose при старте контейнера выполняются `migrate` и `db:seed`. Сиды написаны идемпотентно: пользователи — через `updateOrCreate` по email; заявки создаются только если таблица заявок пуста, чтобы повторный запуск не дублировал данные.

6. **Минимальный UI без сборки фронта**  
   Используются Blade-шаблоны и встроенные в layout стили, без обязательной сборки Vite/Node для быстрого запуска и проверки. При необходимости можно подключить Vite поверх существующих шаблонов.

7. **Middleware `role` для разграничения доступа**  
   Доступ к панели диспетчера и мастера ограничен middleware `EnsureUserRole` по параметру роли (`dispatcher` / `master`), чтобы исключить переход по URL от имени другой роли.
